# 구현계획서: 윈도우 파일 연결(File Association) 메커니즘

## 1. 배경 및 목적

현재 Onrivi Author는 Electron 기반 데스크톱 앱이지만, 윈도우 탐색기에서 `.md` 파일을 더블클릭해도 앱이 실행되지 않거나 실행되어도 해당 파일이 열리지 않는다. 이 계획서는 윈도우 파일 연결을 통해 `.md` 파일 더블클릭 시 Onrivi Author가 해당 파일을 직접 열도록 하는 전 과정을 정의한다.

## 2. 변경 대상 파일

| # | 파일 | 역할 | 변경 방식 |
|---|------|------|-----------|
| ① | `main.js` | Electron 메인 프로세스 | `process.argv` 파싱 + `second-instance` 확장 + IPC 송신 |
| ② | `preload.js` | 컨텍스트 브릿지 | `onReceiveFile` 리스너 노출 |
| ③ | `frontend/src/app/page.tsx` | 프론트엔드 관제탑 | IPC 수신 `useEffect` 추가 |
| ④ | `package.json` (루트) | electron-builder 설정 | `build.win.fileAssociations` 명세 추가 |

## 3. 상세 변경 명세

### 3.① `main.js` — 메인 프로세스 인수 가로채기

**현재 상태:** `second-instance` 핸들러가 존재하나 `commandLine`에서 `.md` 경로를 추출하지 않음. `process.argv`를 전혀 읽지 않음.

**변경 사항:**

#### 3.①-1. `filePathToOpen` 버퍼 변수 추가 (25행 근처)

```javascript
let mainWindow;
let activePort = 4000;
let filePathToOpen = null; // ← 추가: 더블클릭으로 전달된 .md 경로 임시 저장
```

#### 3.①-2. `second-instance` 핸들러에 파일 경로 추출 로직 추가 (35~40행)

```javascript
app.on('second-instance', (event, commandLine, workingDirectory) => {
  if (mainWindow) {
    if (mainWindow.isMinimized()) mainWindow.restore();
    mainWindow.focus();

    // 파일 경로 추출: commandLine에서 .md로 끝나는 첫 인자 탐색
    const fileArg = commandLine.find(arg => {
      const lower = arg.toLowerCase();
      return lower.endsWith('.md') || lower.endsWith('.markdown');
    });
    if (fileArg && fs.existsSync(fileArg)) {
      mainWindow.webContents.send('open-external-md', fileArg);
    }
  }
});
```

#### 3.①-3. `checkFileArgument()` 함수 및 호출 추가

앱 초기 기동 시 `process.argv`를 검사하는 함수를 `app.whenReady()` 이전에 정의하고, `createWindow` 이후 `did-finish-load` 이벤트에서 IPC로 전달한다.

**함수 정의 (42행 근처, `getFreePort` 앞):**

```javascript
// 최초 실행 시 커맨드라인 인수에 .md 파일이 있는지 검사
function checkFileArgument() {
  const fileArg = process.argv.find(arg => {
    const lower = arg.toLowerCase();
    return lower.endsWith('.md') || lower.endsWith('.markdown');
  });
  if (fileArg && fs.existsSync(fileArg)) {
    filePathToOpen = fileArg;
  }
}
```

**`createWindow` 내부, `loadFile`/`loadURL` 이후 (111행 근처):**

```javascript
mainWindow.webContents.on('did-finish-load', () => {
  if (filePathToOpen) {
    mainWindow.webContents.send('open-external-md', filePathToOpen);
    filePathToOpen = null;
  }
});
```

**`app.on('ready')` 내부, `createWindow` 호출 전 (190행 근처):**

```javascript
app.on('ready', async () => {
  // ... protocol.handle('media', ...) 등록 ...

  checkFileArgument(); // ← 추가
  createWindow(activePort);
});
```

#### 3.①-4. 주의사항

- `process.argv[0]`는 Electron 실행 파일 경로, `process.argv[1]`부터 실제 인자.
- `.md` 아닌 `.markdown` 확장자도 함께 처리.
- `did-finish-load`는 `loadFile`/`loadURL` 모두에서 발생하므로 dev 모드와 production 모두 동작.
- 개발 모드(dev)에서는 `npm run dev` 또는 `electron .`로 실행 시 `process.argv`에 불필요한 인자가 섞일 수 있으므로 `.md`/`.markdown` 확장자 필터 필수.

### 3.② `preload.js` — `onReceiveFile` 리스너 노출

**현재 상태:** 메뉴 이벤트 리스너만 존재. `onReceiveFile` 없음.

**변경 사항:**

`removeListeners` 메서드 위에 다음 항목 추가:

```javascript
// 윈도우 탐색기 더블클릭 → 외부 .md 파일 열기 요청 수신
onReceiveFile: (callback) => {
  const handler = (_event, filePath) => callback(filePath);
  ipcRenderer.on('open-external-md', handler);
  // 구독 해제 함수 반환 (React useEffect cleanup에서 사용)
  return () => {
    ipcRenderer.removeListener('open-external-md', handler);
  };
},
```

**`removeListeners`에도 cleanup 추가:**

```javascript
removeListeners: () => {
  ipcRenderer.removeAllListeners('menu:new-file');
  ipcRenderer.removeAllListeners('menu:open-file');
  ipcRenderer.removeAllListeners('menu:save-file');
  ipcRenderer.removeAllListeners('menu:save-file-as');
  ipcRenderer.removeAllListeners('open-external-md'); // ← 추가
}
```

### 3.③ `page.tsx` — 프론트엔드 IPC 수신부

**현재 상태:** `handleFileOpenByPath(resolvedPath)` 함수가 이미 존재(1296행). `electronAPI`를 통한 메뉴 이벤트 수신 `useEffect`도 존재(994행). 그러나 `onReceiveFile` 리스너가 없음.

**변경 사항:**

기존 메뉴 이벤트 수신 `useEffect`(994~1006행)에 `onReceiveFile` 구독을 병합:

```typescript
useEffect(() => {
  if (typeof window !== 'undefined' && (window as any).electronAPI) {
    const api = (window as any).electronAPI;
    api.onNewFileRequested(() => handlers.newFile());
    api.onOpenFileRequested(() => handlers.openFolder());
    api.onSaveFileRequested(() => handlers.save());
    api.onSaveFileAsRequested(() => handlers.saveAs());

    // ★ 윈도우 탐색기 더블클릭 → 외부 .md 파일 열기
    let unsubscribeReceiveFile: (() => void) | undefined;
    if (api.onReceiveFile) {
      unsubscribeReceiveFile = api.onReceiveFile((filePath: string) => {
        handleFileOpenByPath(filePath);
      });
    }

    return () => {
      api.removeListeners();
      if (unsubscribeReceiveFile) unsubscribeReceiveFile();
    };
  }
}, [mounted, content, currentFileNode]);
```

#### 3.③-1. 핵심 의존성

`handleFileOpenByPath`는 `fileList`, `handleFileClick`, `workspaceType` 등에 의존한다. 이미 동일 `useEffect` 내에서 `handlers` 객체를 통해 접근 중이므로 추가 의존성 처리는 불필요. 단, `handleFileOpenByPath`가 `useCallback`으로 감싸져 있지 않다면 `useEffect` 내에서 직접 호출하지 않고 `handlers`와 동일한 패턴으로 참조할 것.

### 3.④ `package.json` (루트) — electron-builder 파일 연결 등록

**현재 상태:** `build.win`에 `target: ["nsis", "portable"]`만 존재. `fileAssociations` 없음.

**변경 사항:**

`build.win`에 `fileAssociations` 배열 추가:

```json
"win": {
  "target": ["nsis", "portable"],
  "icon": "frontend/public/icon.png",
  "fileAssociations": [
    {
      "ext": "md",
      "name": "Markdown Document",
      "description": "온리비 어서 마크다운 문서",
      "role": "Editor"
    },
    {
      "ext": "markdown",
      "name": "Markdown Document",
      "description": "온리비 어서 마크다운 문서",
      "role": "Editor"
    }
  ]
}
```

#### 3.④-1. 동작 원리

- electron-builder가 NSIS 인스톨러 생성 시, 위 명세를 읽어 Windows 레지스트리에 `HKEY_CLASSES_ROOT\.md` 및 `HKEY_CLASSES_ROOT\.markdown` 항목을 등록한다.
- 등록 후 `.md` 파일 우클릭 → "연결 프로그램"에 "Onrivi Author"가 나타난다.
- 더블클릭 시 `OnriviAuthor.exe "%1"` 형태로 실행되며, `%1`이 `process.argv`로 전달된다.
- `portable` target은 레지스트리를 수정하지 않으므로 파일 연결 기능을 사용하려면 **NSIS 설치 버전**으로 배포해야 한다.

## 4. 데이터 흐름

```
[Windows 탐색기] .md 파일 더블클릭
       │
       ▼
[OS] "OnriviAuthor.exe" "D:\문서\example.md" 실행
       │
       ├── (첫 실행) process.argv = ["...OnriviAuthor.exe", "D:\문서\example.md"]
       │       → checkFileArgument() → filePathToOpen = "D:\문서\example.md"
       │       → createWindow() → did-finish-load
       │       → IPC send('open-external-md', "D:\문서\example.md")
       │
       └── (이미 실행 중) second-instance 이벤트
               → commandLine.find(".md") → IPC send('open-external-md', path)
                      │
                      ▼
               [preload.js] ipcRenderer.on('open-external-md')
                      │
                      ▼
               [page.tsx] onReceiveFile 콜백
                      │
                      ▼
               handleFileOpenByPath(filePath)
                      │
                      ├── fileList에서 경로 탐색 → 있으면 handleFileClick
                      └── 없으면 dummy FileNode 생성 → handleFileClick
                             │
                             ▼
                           electronAPI.readFromPath(filePath)
                             │
                             ▼
                           에디터에 콘텐츠 로드
```

## 5. 테스트 계획

### 5.1 개발 모드 테스트

| 단계 | 동작 | 예상 결과 |
|------|------|-----------|
| 1 | `electron .` 실행 | 앱 정상 기동, 추가 파일 안 열림 |
| 2 | `electron . "C:\test\sample.md"` 실행 | 앱 기동 후 sample.md 내용이 에디터에 로드됨 |
| 3 | 앱 실행 중, `electron . "C:\test\another.md"` 실행 | 기존 창 포커싱 → another.md 로드 |
| 4 | `.markdown` 파일로 2,3 반복 | 동일하게 동작 |

### 5.2 빌드 및 설치 테스트

| 단계 | 동작 | 예상 결과 |
|------|------|-----------|
| 1 | `npm run dist` | NSIS 인스톨러 생성 |
| 2 | 인스톨러로 설치 | `.md` 파일 아이콘이 Onrivi Author 아이콘으로 변경 |
| 3 | `.md` 파일 더블클릭 | 앱 실행 → 파일 열림 |
| 4 | 앱 켜진 상태에서 `.md` 더블클릭 | 기존 창 포커싱 → 해당 파일 열림 |

### 5.3 예외 케이스

| 케이스 | 처리 |
|--------|------|
| 존재하지 않는 경로 | `fs.existsSync()`로 필터 → IPC 미전송 |
| `.md`가 아닌 파일 | 확장자 필터로 차단 |
| 경로에 한글/공백 포함 | `.normalize('NFC')` + OS가 자동 quote 처리 |
| 매우 긴 경로 | Windows MAX_PATH(260자) 확인, `\\?\` 프리픽스 필요 시 추가 검토 |
| 파일 내용 읽기 실패 | `handleFileOpenByPath` 내부 `showToast`로 에러 표시 |

### 6.2 부분 롤백

## 7. 위험 요소 및 대응

| 위험 | 영향 | 확률 | 대응 |
|------|------|------|------|
| `second-instance`에서 `commandLine`이 예상과 다른 포맷 | 파일 열기 실패 | 낮음 | `.md` 확장자 필터 + 로그 출력 |
| `did-finish-load`가 너무 빨리 발생 | IPC 전달 실패 | 낮음 | `filePathToOpen` 버퍼 변수로 보장 |
| NSIS 설치 후 레지스트리 미등록 | 파일 연결 안 됨 | 중간 | `electron-builder` 버전 확인, `"autoUpdater"` 없이도 동작하는지 확인 |
| `portable` 버전 사용 시 파일 연결 안 됨 | 기능 무효 | 높음 | 문서에 NSIS 설치 버전 필수임을 명시 |
| macOS/Linux 미지원 | 크로스 플랫폼 불가 | 높음 | 이번 계획은 Windows 전용. macOS는 `app.on('open-file', ...)`로 별도 구현 필요. |

## 8. 향후 확장 (v2)

- **macOS 지원:** `app.on('open-file', (event, path) => { ... })` 이벤트 리스너 추가.
- **커스텀 프로토콜:** `onriviauth://open?path=...` 형태의 딥 링크 지원.
- **다중 파일:** 여러 `.md` 파일을 한 번에 더블클릭했을 때 모두 탭으로 열기.
- **최근 파일 목록:** Windows JumpList에 최근 연 파일 표시.
